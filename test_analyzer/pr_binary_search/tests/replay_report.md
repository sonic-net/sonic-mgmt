## The two bad_commit false-skips

Both false-skips concern the **same test** (`test_critical_process_monitoring /
t0_checker`), which is the hardest-to-bisect test in the 30-day window —
it routinely produces 6-8 consecutive `unknown` outcomes before
converging on a real `bad_commit`.

| Filter | When | Why it fires | Production impact |
|---|---|---|---|
| `F1_EXISTING_RESULT` | 2026-05-19 03:21 UTC | Previous bad_commit `59b5f7…` was logged ~1 hour earlier | Eliminated in prod: dispatcher dedup collapses two failures of the same test within 1 h into a single entry **before** the filter runs |
| `F3_CIRCUIT_BREAKER` | 2026-05-14 16:40 UTC | 7 consecutive `unknown` searches in the prior 60 d | Real recall loss — this is the trade-off F3 makes |

The F3 case is a real recall regression we accept because the test in
question is a candidate for either the F0 blocklist or for F4 (which
catches the pre-existing-failure pattern) once F4 is validated.

## Tuning history & trade-offs

This filter has been tuned through three iterations.  Numbers in
parentheses are from the committed 2026-05-26 snapshot.

| Configuration | Naive recall | Noise red. | F3 fires (unk + bad) | Notes |
|---|---:|---:|---:|---|
| v0 — F1 30d / F2 90 %/10 / F3 2-in-14d | 37.5% | 65.4% | many | Way too aggressive; original |
| v1 — F1 7d most-recent-only / F2 98 %/200 / F3 8-in-7d | 87.5% | 30.8% | 0 + 0 | Tight recall but F3 effectively unreachable |
| **v2 (current)** — F1 30d w/ repeat-rule / F2 98 %/200 / F3 7-in-60d | **75.0%** | **33.3%** | 1 + 1 | F3 actually fires; F1 broadens to catch repeat bad commits |

### Why v2's broader filters?

The v1 thresholds attracted the following review findings (each addressed
in this commit):

- **F1 missed the "found bad_commit but still failing" case.**  v1's 7-day
  lookback was shorter than the typical inter-failure gap on a fix-not-yet-
  merged regression (10-21 days).  v2 extends lookback to 30 days and adds a
  **repeat-bad-commit rule**: same `BadCommit` seen ≥ 2× in the window →
  skip even if the most recent prior was an `unknown`.
- **F3 was unreachable in production.**  v1 required 8 consecutive unknowns
  inside 7 days; real per-key search rates are ~2 / 14 d, so this fired
  0 times.  v2 widens to 60 days and lowers N to 7.  This trades 12.5 %
  naive recall for a filter that actually fires.
- **F4 had a `span_days` bug** that classified a previously-clean,
  newly-failing test as pre-existing (it used `min/max(StartTime)` over
  *all* runs).  v2 uses `min/max(StartTime)` over **failing** runs only.
  Covered by the unit test `test_f4_passes_when_previously_clean_then_newly_failing`.
- **F4 not yet validated against historical data.**  v2 disables F4 in the
  default enabled set; opt in once a per-test replay dataset is captured.
- **F2 docstring was out of sync** ("90 %" while threshold is 0.98) — fixed.

## New metric: "repeat bad_commit findings avoided"

The replay tool now reports `repeat_bad_commit_avoided` — the number of
historical searches whose outcome was a `bad_commit` we **had already
identified** in the window.  These are noise we couldn't see in the
unknown count: bisecting the same regression a second time costs the
same as bisecting an unknown, but the result looks like a "find" so it's
invisible in naive recall / noise-reduction.

`effective_recall` rewrites recall to exclude these rediscoveries from
the denominator: it's the fraction of *new* regressions caught.

The 2026-05-26 snapshot has 0 such rediscoveries (every `bad_commit` in
the window is distinct), so naive and effective recall coincide.  The
metric will become non-zero on snapshots that contain a "fix wasn't
merged" episode, and is the right number to optimise on the next round
of tuning.

```bash

cd test_analyzer/pr_binary_search

    --snapshot tests/replay_kusto_data_2026-05-26.json

# Or query Kusto live (recommended after any threshold change)
export ACCESS_TOKEN=$(az account get-access-token \
    --resource https://api.kusto.windows.net --query accessToken -o tsv)
```
